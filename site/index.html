Why shielded payments
Agents leak payment data

When AI agents pay on public chains, every transaction reveals who called what API, how often, and how much they spent. That's a competitive intelligence goldmine. Zcash fixes this.

Usage fingerprinting

Public payments let competitors see exactly which APIs your agent uses, how frequently, and spending patterns.

Price discrimination

When services can see your total spend across APIs, they can adjust pricing based on your usage profile.

Encrypted verification

Servers verify payments using an Orchard incoming viewing key. Only the recipient can decrypt and confirm — no one else.

Server

import { Mppx } from 'mppx/server'
import { zcash } from 'zimppy-ts/server'

const mppx = Mppx.create({
  methods: [await zcash({ wallet: 'server' })],
})

const result = await mppx.charge({
  amount: '42000',
  currency: 'zec',
})(request)

if (result.status === 402) return result.challenge
return result.withReceipt(Response.json({ data }))
use mpp::server::Mpp;
use zimppy_rs::ZcashChargeMethod;

let charge = ZcashChargeMethod::new(
  &rpc_endpoint, &address, &orchard_ivk,
);
let mpp = Mpp::new(charge, "my-api", &secret);

// axum handler:
match mpp.verify_credential(&credential).await {
  Ok(receipt) => (
    [(header::from_static("payment-receipt"),
      receipt.to_header())],
    Json(data),
  ).into_response(),
  Err(_) => issue_402(&mpp),
}

How it works
Standard HTTP 402,
shielded underneath

Same MPP flow every agent already speaks. The server returns a 402 challenge, the agent pays with shielded ZEC, retries with proof, and gets the response. One HTTP round-trip, fully private.

Sessions
Deposit once, instant bearer requests
1. Open

Deposit ZEC

Client sends a shielded deposit and a bearer secret. Server creates the session.

on-chain
2. Bearer

Instant requests

Every request uses the bearer token. Server deducts from balance. No chain needed.

instant
3. Top Up

Add more ZEC

Balance low? Send another shielded deposit. Session continues without interruption.

on-chain
4. Close

Get refund

Close the session. Server refunds unused balance back to the client's address.

on-chain 
import { Mppx } from 'mppx/server'
import { zcash } from 'zimppy-ts/server'

const mppx = Mppx.create({
  methods: [await zcash.session({
    wallet: 'server',
    suggestedDeposit: 100000,
    unitType: 'request',
  })],
})

const result = await mppx.session({
  amount: '5000',
  unitType: 'request',
})(request)

if (result.status === 402) return result.challenge
return result.withReceipt(Response.json({ data }))
import { Mppx } from 'mppx/client'
import { zcash } from 'zimppy-ts/client'

const session = zcash.session({ wallet: 'default' })

const mppx = Mppx.create({ methods: [session] })

// First fetch deposits, subsequent fetches use bearer
const r1 = await mppx.fetch(url)  // deposit on-chain
const r2 = await mppx.fetch(url)  // instant bearer
const r3 = await mppx.fetch(url)  // instant bearer

// Done — close and get refund
session.close()
await mppx.fetch(url)
